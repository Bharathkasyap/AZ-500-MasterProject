## 1. Microsoft Defender for Cloud

### What It Is
Microsoft Defender for Cloud (formerly Azure Security Center) is a Cloud Security Posture Management (CSPM) and Cloud Workload Protection Platform (CWPP) for Azure, hybrid, and multicloud environments.

### Key Functions

| Function | Description |
|---|---|
| **CSPM** | Assess security posture, identify misconfigurations, provide recommendations |
| **CWPP** | Protect workloads with threat detection and response capabilities |
| **Regulatory compliance** | Track compliance against standards (CIS, PCI-DSS, NIST, ISO 27001) |

### Defender for Cloud Tiers

| Tier | Features |
|---|---|
| **Free (Foundational CSPM)** | Continuous assessment, secure score, basic recommendations |
| **Defender CSPM** | Attack path analysis, cloud security graph, governance |
| **Workload protections** | Defender plans for specific resource types (per resource/month) |

### Defender Plans

| Defender Plan | Protects |
|---|---|
| Defender for Servers | VMs (Azure, AWS, GCP, on-premises) |
| Defender for Storage | Azure Storage accounts |
| Defender for SQL | Azure SQL, SQL Server on VMs, open-source DBs |
| Defender for Containers | AKS, container registries, Kubernetes |
| Defender for App Service | Azure App Service |
| Defender for Key Vault | Azure Key Vault |
| Defender for DNS | DNS-layer threats |
| Defender for Resource Manager | ARM operations monitoring |

### Secure Score
- A numeric score (0–100%) representing your security posture
- Increases when you implement security recommendations
- Used to track improvement over time
- Higher score = better security posture

### Security Recommendations
- Generated by assessing resources against security policies
- Each recommendation shows: affected resources, severity (High/Medium/Low), potential score increase
- **Quick fix** — One-click remediation for some recommendations

### Security Alerts
- Real-time alerts about threats to your Azure resources
- Severity levels: High, Medium, Low, Informational
- Exported to Microsoft Sentinel, Log Analytics, or SIEM

### Regulatory Compliance Dashboard
- Map your Azure configuration to compliance requirements
- Supports: Azure Security Benchmark, CIS, PCI-DSS, NIST SP 800-53, ISO 27001, SOC TSP
- Compliance score per standard

> **Exam tip:** Defender for Cloud's secure score is based on implementing recommendations. Workload protection plans require paid Defender plans to be enabled.

## 2. Microsoft Defender for Servers (Just-in-Time VM Access)

### Just-in-Time (JIT) VM Access

JIT VM Access reduces attack surface by locking down inbound traffic to VMs and opening ports only when needed, for a limited time, to authorized users.

### How JIT Works

1. Admin enables JIT on a VM (defines rules: which ports, max time, source IPs)
2. Default JIT policy blocks ports like RDP (3389), SSH (22), WinRM (5985/5986)
3. User requests access via Azure Portal, CLI, or API
4. Defender for Cloud evaluates the request (RBAC check)
5. Approved: NSG rule is added allowing the specific user's IP for the specified time
6. After the time expires: NSG rule is automatically removed

### JIT Requirements
- Microsoft Defender for Servers Plan 2 must be enabled
- VM must have a public IP or be behind an Azure Firewall
- User needs `Microsoft.Security/locations/jitNetworkAccessPolicies/initiate/action` permission

### Defender for Servers Features

| Feature | Description |
|---|---|
| **JIT VM Access** | Controlled port access |
| **Adaptive Application Controls** | Allow-list for applications running on VMs |
| **File Integrity Monitoring (FIM)** | Detect changes to critical files and registry |
| **Vulnerability Assessment** | Integrated Qualys or Microsoft TVM scanner |
| **EDR Integration** | Microsoft Defender for Endpoint integration |

### Adaptive Application Controls
- Machine learning-based allow-listing
- Recommends which applications should be allowed to run on VMs
- Alerts when unexpected applications are executed

> **Exam tip:** JIT requires Defender for Servers. Know the workflow: request → RBAC check → NSG rule added → time expires → rule removed.

## 3. Azure Key Vault

### What It Is
Azure Key Vault is a cloud service for securely storing and accessing secrets, keys, and certificates.

### Three Types of Objects

| Type | Examples |
|---|---|
| **Secrets** | Connection strings, passwords, API keys, storage account keys |
| **Keys** | RSA or EC cryptographic keys for encryption/signing operations |
| **Certificates** | X.509 certificates (managed lifecycle: creation, renewal, export) |

### Key Vault Tiers

| Tier | Hardware Protection | Cost |
|---|---|---|
| **Standard** | Software-protected keys | Lower |
| **Premium** | HSM-protected keys (FIPS 140-2 Level 2) | Higher |
| **Managed HSM** | Dedicated single-tenant HSM (FIPS 140-2 Level 3) | Highest |

### Access Control Models

#### 1. Vault Access Policy (legacy)
- Set per vault; applies to all objects in vault
- Permission levels: Get, List, Set, Delete, Backup, Restore, Recover, Purge
- Separate permissions for keys, secrets, and certificates

#### 2. Azure RBAC for Key Vault (recommended)
- Uses Azure RBAC for granular, object-level permissions
- Supports fine-grained control (e.g., access to specific secrets only)

| Built-in Role | Permissions |
|---|---|
| Key Vault Administrator | All operations on all object types |
| Key Vault Certificates Officer | Manage certificates (not keys/secrets) |
| Key Vault Crypto Officer | Manage keys |
| Key Vault Secrets Officer | Manage secrets |
| Key Vault Reader | Read metadata (not secret values) |
| Key Vault Crypto User | Use keys for cryptographic operations |
| Key Vault Secrets User | Read secret values |

### Key Vault Soft Delete and Purge Protection

| Feature | Description |
|---|---|
| **Soft Delete** | Deleted objects recoverable for 7–90 days (enabled by default) |
| **Purge Protection** | Prevents permanent deletion during retention period; cannot be disabled once enabled |

> **Exam tip:** Enable Purge Protection to prevent accidental or malicious permanent deletion. Once enabled, it cannot be turned off.

### Key Vault Firewall and Private Endpoint
- Restrict access to specific IP addresses or VNet subnets
- Use Private Endpoint to access Key Vault over private network

### Key Rotation
- Manual rotation: Update secret value; applications get new version automatically
- Automatic rotation for:
  - Storage account keys (Key Vault manages rotation)
  - Certificates (auto-renewal before expiry)
  - Keys (rotation policy with configurable frequency)

### HSM-Protected Keys
- Keys created with HSM protection never leave the HSM in plaintext
- Used for sensitive cryptographic operations
- Bring Your Own Key (BYOK) — Import existing HSM-protected keys

> **Exam tip:** Understand the difference between Vault Access Policy and Azure RBAC. Know when to use Managed HSM vs. Premium Key Vault.

## 4. Azure Disk Encryption

### Encryption Options for Azure VMs

| Method | Description | Key Storage |
|---|---|---|
| **Azure Disk Encryption (ADE)** | Uses BitLocker (Windows) or DM-Crypt (Linux) | Keys in Azure Key Vault |
| **Server-Side Encryption (SSE)** | Encrypts data at rest on Azure Storage | Platform-managed or customer-managed keys |
| **Encryption at host** | Encrypts temp disk and cache; SSE on OS/data disks | Customer-managed keys in Key Vault |
| **Confidential disk encryption** | Encrypts with keys managed by TPM in VM | Guest-managed, not accessible to Azure |

### Azure Disk Encryption (ADE)

- **Windows:** Uses BitLocker
- **Linux:** Uses DM-Crypt
- Encryption keys and secrets stored in Azure Key Vault
- Requires Key Vault in the same region as the VM
- Key Vault must have the Azure Disk Encryption access policy enabled

### Customer-Managed Keys (CMK) for SSE

- Keys stored in Azure Key Vault or Managed HSM
- Allows you to control key rotation, disable, and revoke access
- Disk Encryption Set (DES) — a resource that maps your CMK to managed disks

### Double Encryption
- Encrypts data twice: once with platform-managed key + once with customer-managed key
- Available for managed disks, snapshots, and images

> **Exam tip:** ADE encrypts at the OS level (BitLocker/DM-Crypt). SSE encrypts at the storage layer. Both can be used together.

## 5. Storage Account Security

### Storage Account Security Layers

| Layer | Mechanism |
|---|---|
| **Authentication** | Storage account keys, SAS tokens, Azure AD (RBAC), Managed Identities |
| **Authorization** | Azure RBAC (data plane), ACLs (ADLS Gen2) |
| **Encryption** | Azure Storage Service Encryption (SSE) at rest; HTTPS in transit |
| **Network** | Firewall rules, VNet service endpoints, Private Endpoints |
| **Threat protection** | Microsoft Defender for Storage |

### Authentication Methods

#### Storage Account Keys
- Two 512-bit keys per account (key1, key2)
- Full access to all data in the account
- Should be rotated regularly and stored in Key Vault
- **Risk:** If leaked, attacker has full access — rotate immediately

#### Shared Access Signatures (SAS)
Grants limited, time-bound access to storage resources.

| SAS Type | Scope | Key Used |
|---|---|---|
| **Account SAS** | Multiple services in the account | Storage account key |
| **Service SAS** | One service (Blob, Queue, Table, or File) | Storage account key |
| **User Delegation SAS** | Blob storage only | Azure AD credentials (most secure) |

**SAS Parameters:**
- Service (blob, file, queue, table)
- Permissions (read, write, delete, list, create)
- Expiry date/time
- Allowed IP ranges
- Required protocol (HTTPS only recommended)

> **Exam tip:** User Delegation SAS is the most secure SAS type because it uses Azure AD credentials, not the storage account key.

### Storage Account Firewall
- Default: Allow all networks
- Configure to: Allow specific VNets and IP addresses only
- Enable "Trusted Microsoft services" for Azure services that need access

### Secure Transfer Required
- Forces all connections to use HTTPS (deny HTTP)
- Enabled by default on new storage accounts
- Critical security setting — always enable

### Microsoft Defender for Storage

| Alert Type | Description |
|---|---|
| Unusual access patterns | Access from anonymous IPs, unusual locations |
| Malware detection | Detect malware uploaded to blob storage |
| Sensitive data access | Access to sensitive data (PII) detection |
| Data exfiltration | Unusual amounts of data being extracted |

### Immutable Blob Storage
- WORM (Write Once, Read Many) policies
- **Time-based retention policy** — blobs cannot be deleted/modified for a period
- **Legal hold** — blobs locked until legal hold is cleared
- Compliance requirements: SEC 17a-4, FINRA, CFTC

### Azure Storage Encryption

- All data encrypted at rest by default using AES-256
- **Platform-managed keys (PMK)** — Microsoft manages the keys
- **Customer-managed keys (CMK)** — Stored in Key Vault; customer controls rotation/revocation

> **Exam tip:** Know the difference between SAS types. User Delegation SAS doesn't use the storage account key.

## 6. Azure SQL Database Security

### SQL Database Security Layers

| Layer | Feature |
|---|---|
| **Authentication** | SQL authentication, Azure AD authentication |
| **Authorization** | Database-level roles, object-level permissions |
| **Encryption (transit)** | TLS encryption enforced |
| **Encryption (rest)** | Transparent Data Encryption (TDE) |
| **Threat protection** | Microsoft Defender for SQL, Advanced Threat Protection |
| **Auditing** | SQL Auditing to Storage, Log Analytics, or Event Hub |
| **Network** | Firewall rules, Private Endpoints, VNet Service Endpoints |
| **Data masking** | Dynamic Data Masking (DDM) |

### Transparent Data Encryption (TDE)

- Encrypts the database, backups, and transaction log files at rest
- Enabled by default on all Azure SQL databases
- **Service-managed key** — Azure manages the key
- **Bring Your Own Key (BYOK)** — Customer-managed key in Azure Key Vault (TDE Protector)

### Azure AD Authentication for SQL
- Use Azure AD identities instead of SQL logins
- Supports MFA, Conditional Access, and managed identities
- More secure than SQL authentication (no passwords to manage)
- Set an Azure AD admin for the SQL server

### Microsoft Defender for SQL

| Feature | Description |
|---|---|
| **Vulnerability Assessment** | Periodic scan for SQL misconfigurations and vulnerabilities |
| **Advanced Threat Protection** | Real-time alerts for SQL injection, unusual access, etc. |

**Advanced Threat Protection alerts:**
- SQL injection detection
- SQL injection vulnerability
- Access from unusual location
- Access from unusual Azure data center
- Access from potentially harmful application

### SQL Auditing
- Records database events to an audit log
- Destinations: Storage account, Log Analytics workspace, Event Hub
- Required for many compliance frameworks
- Audit log includes: principal, action, object, success/failure

### Dynamic Data Masking (DDM)
- Masks sensitive data in query results for non-privileged users
- Data remains unchanged in the database — only the view is masked
- Built-in masking functions: Email, Default, Random, Custom string, Credit card

**Example:** Email `user@example.com` → masked as `uXXX@XXXX.com`

### Row-Level Security (RLS)
- Control which rows a user can read or modify
- Uses security predicates (filter and block predicates)
- Transparent to the application

### Always Encrypted
- Client-side encryption — the database engine never sees plaintext
- Keys managed by the client application
- Supports: Deterministic encryption (for equality comparisons) and Randomized encryption (stronger)

> **Exam tip:** TDE encrypts at rest; Always Encrypted protects from DBA access. DDM only masks display, not actual data.

## 7. Container Security (AKS and ACR)

### Azure Container Registry (ACR) Security

| Feature | Description |
|---|---|
| **Authentication** | Azure AD, managed identity, admin account (disabled by default) |
| **Authorization** | RBAC roles: AcrPull, AcrPush, AcrDelete, Owner |
| **Image scanning** | Microsoft Defender for Containers — scans for CVEs |
| **Content trust** | Image signing with Docker Content Trust |
| **Private Endpoint** | Access ACR without public internet |
| **Geo-replication** | Replicate images across regions |

**ACR RBAC Roles:**

| Role | Permissions |
|---|---|
| AcrPull | Pull images |
| AcrPush | Push and pull images |
| AcrDelete | Delete images |
| AcrImageSigner | Sign images |

### Azure Kubernetes Service (AKS) Security

| Area | Security Control |
|---|---|
| **Authentication** | Azure AD integration, service accounts |
| **Authorization** | Kubernetes RBAC, Azure RBAC for AKS |
| **Network policies** | Calico or Azure network policies to control pod communication |
| **Secrets management** | AKS + Key Vault (Azure Key Vault Provider for Secrets Store CSI Driver) |
| **Node security** | OS patching, CIS-hardened node images |
| **Image security** | Microsoft Defender for Containers, image scanning |
| **Cluster access** | Private cluster (no public API server endpoint) |

### AKS Private Cluster
- API server not accessible from internet
- Access via VNet integration or Azure Private Link
- Requires private endpoint for API server

### Pod Identity and Workload Identity
- **Workload Identity** (recommended) — Uses Azure AD Workload Identity to grant pod access to Azure resources without managing credentials
- Replaces deprecated AAD Pod Identity

> **Exam tip:** Disable ACR admin account. Use AcrPull managed identity for AKS to pull images from ACR.

## 8. Azure App Service Security

### App Service Authentication (Easy Auth)
- Built-in authentication/authorization module
- Supports: Azure AD, Microsoft, Facebook, Google, Twitter, Apple, OpenID Connect
- No code changes required in the application

### App Service TLS/HTTPS
- Enforce HTTPS-only (redirect HTTP to HTTPS)
- Configure minimum TLS version (1.2 recommended)
- Custom domain SSL certificates via App Service Certificates or Key Vault

### App Service Network Security

| Feature | Description |
|---|---|
| **Access restrictions** | IP-based allow/deny rules for inbound traffic |
| **Service Endpoints** | Restrict access to specific VNet subnets |
| **Private Endpoints** | Access App Service over private IP |
| **VNet Integration** | Allow App Service to access resources in a VNet (outbound) |

### App Service Environment (ASE)
- Fully isolated, dedicated App Service environment
- Deployed in your VNet
- No public IP (internal ASE)
- Supports compliance requirements needing full network isolation

### Managed Identity for App Service
- Enable managed identity for App Service
- Access Key Vault secrets, Storage, SQL without credentials in code

> **Exam tip:** VNet Integration allows App Service to reach resources IN your VNet (outbound). Private Endpoint allows resources to reach App Service (inbound).
