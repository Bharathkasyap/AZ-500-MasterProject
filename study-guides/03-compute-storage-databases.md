## 1. Microsoft Defender for Cloud

**Microsoft Defender for Cloud** (formerly Azure Security Center + Azure Defender) is a unified cloud security posture management (CSPM) and cloud workload protection platform (CWPP).

### Two Pillars

| Pillar | Description |
|--------|-------------|
| **CSPM (Cloud Security Posture Management)** | Assess security posture, find misconfigurations, provide Secure Score |
| **CWPP (Cloud Workload Protection Platform)** | Threat detection and response for workloads (VMs, containers, DBs, etc.) |

### Defender Plans (per resource type)

| Plan | Protects |
|------|---------|
| Defender for Servers | VMs (Azure, AWS, GCP, on-prem via Arc) |
| Defender for Storage | Blob, File, ADLS Gen2 — detect malware, suspicious activity |
| Defender for SQL | Azure SQL, SQL on VMs, SQL managed instances |
| Defender for Containers | AKS, container registries, Kubernetes clusters |
| Defender for Key Vault | Unusual access, enumeration attempts |
| Defender for App Service | Detect attacks on App Service apps |
| Defender for Resource Manager | Detect unusual ARM operations |
| Defender for DNS | Detect suspicious DNS queries |
| Defender CSPM (paid) | Advanced CSPM: attack path analysis, cloud security graph |

### Secure Score

- Numeric score (0–100%) representing your security posture
- Divided into **security controls** (groups of related recommendations)
- Each control has a max score contribution; completing all recommendations in a control earns the full points
- **Recommendations** are actionable items; each has a severity (High/Medium/Low), affected resources, and remediation steps

```
Secure Score Formula:
  Score = (Current score of all controls) / (Max score of all controls) × 100
```

### Security Alerts

- Generated by Defender plans when suspicious activity is detected
- Severity: High, Medium, Low, Informational
- Can trigger **automation** (Logic Apps, Power Automate) for response
- Can be streamed to **Microsoft Sentinel**, **SIEM** (via Continuous Export), or **Event Hubs**

### Regulatory Compliance

- Maps your Azure configuration to compliance standards (PCI DSS, NIST, CIS, ISO 27001, etc.)
- Shows passing/failing assessments per standard
- Requires paid Defender CSPM or specific Defender plans

### Defender for Cloud — Multi-Cloud

- Supports **AWS** and **GCP** via Cloud Connectors
- Onboards non-Azure machines via **Azure Arc**
- Provides unified posture view across clouds

## 2. Azure Key Vault

**Azure Key Vault** is a cloud service for securely storing and accessing secrets, keys, and certificates.

### Three Object Types

| Type | Description | Examples |
|------|-------------|---------|
| **Secrets** | Small sensitive values | Passwords, connection strings, API keys |
| **Keys** | Cryptographic keys | RSA 2048/3072/4096, EC P-256/P-384/P-521 |
| **Certificates** | X.509 certificates with private key | TLS certs, code signing certs |

### HSM Tiers

| Tier | Key Protection |
|------|---------------|
| **Standard** | Software-protected keys |
| **Premium** | HSM-protected keys (FIPS 140-2 Level 2) |
| **Managed HSM** | Single-tenant, FIPS 140-2 Level 3 HSM; customer controls all key material |

### Access Models

#### Option 1: Vault Access Policies (Legacy)

- Per-principal permissions for keys, secrets, and certificates separately
- Permissions: Get, List, Set, Delete, Backup, Restore, Recover, Purge, etc.
- Limited to 1024 access policy entries per vault

#### Option 2: Azure RBAC (Recommended)

- Uses Azure role assignments (integrated with Entra ID)
- Key built-in roles:
  - **Key Vault Administrator**: Full management of all objects
  - **Key Vault Secrets Officer**: Manage secrets (CRUD)
  - **Key Vault Secrets User**: Read secret values
  - **Key Vault Keys Officer**: Manage keys
  - **Key Vault Crypto User**: Use keys (sign/verify/encrypt/decrypt)
  - **Key Vault Certificate Officer**: Manage certificates
  - **Key Vault Reader**: View metadata (not secret values)

> **Exam Tip:** If both Vault Access Policies and RBAC are configured, RBAC takes precedence when RBAC is the selected permission model. Choose one approach consistently.

### Soft Delete & Purge Protection

| Feature | Description |
|---------|-------------|
| **Soft Delete** | Deleted objects retained for 7–90 days (default 90); can be recovered. **Enabled by default** (cannot be disabled). |
| **Purge Protection** | Prevents permanent deletion until retention period ends. Must be enabled explicitly. |

> **Exam Tip:** **Purge Protection** is required for Key Vault keys used with **Customer-Managed Keys (CMK)** for Azure Storage and Disk Encryption to prevent accidental permanent deletion.

### Network Access

- **Firewall and virtual networks**: Restrict Key Vault access to specific IP ranges or VNet subnets.
- **Private Endpoint**: Access Key Vault via private IP in your VNet (recommended for highest security).
- **Trusted Microsoft services**: Allow Azure services that support trusted access to bypass the firewall.

```bash
# Create Key Vault with RBAC and purge protection
az keyvault create \
  --name MyKeyVault \
  --resource-group MyRG \
  --location eastus \
  --sku standard \
  --enable-rbac-authorization true \
  --enable-purge-protection true \
  --retention-days 90

# Store a secret
az keyvault secret set \
  --vault-name MyKeyVault \
  --name "DbConnectionString" \
  --value "Server=myserver;Database=mydb;..."

# Retrieve a secret
az keyvault secret show \
  --vault-name MyKeyVault \
  --name "DbConnectionString" \
  --query value -o tsv
```

### Key Vault Logging

- Diagnostic logs: `AuditEvent` (all API calls to Key Vault)
- Send to Log Analytics, Storage Account, or Event Hub
- Logs include: caller identity, operation, result, key/secret/cert name

### Customer-Managed Keys (CMK)

By default, Azure encrypts data at rest with **Microsoft-managed keys** (MMK). You can bring your own keys using **CMK** stored in Key Vault:
- Azure Storage encryption
- Azure Disk Encryption (with CMK in Key Vault)
- Azure SQL Transparent Data Encryption
- Azure Backup

## 3. Azure Virtual Machine Security

### Azure Disk Encryption (ADE)

Encrypts OS and data disks of Azure VMs:
- **Windows VMs**: Uses **BitLocker**
- **Linux VMs**: Uses **DM-Crypt**
- Keys stored in **Azure Key Vault**
- Encrypts full disk at the OS level (in-VM encryption)

```bash
# Enable ADE on a VM
az vm encryption enable \
  --resource-group MyRG \
  --name MyVM \
  --disk-encryption-keyvault MyKeyVault
```

**ADE vs Server-Side Encryption (SSE):**

| Feature | ADE (Azure Disk Encryption) | SSE (Platform/Server-Side) |
|---------|----------------------------|---------------------------|
| Where encryption happens | Inside the VM (BitLocker/DM-Crypt) | Azure storage platform |
| Boot disk encrypted | ✅ Yes | ✅ Yes (managed disks default) |
| Key stored in | Key Vault | Platform or Key Vault (CMK) |
| Encryption at rest | ✅ | ✅ |
| Encryption in-use/in-transit | ✅ (additional layer) | ❌ |

### Endpoint Protection & Antimalware

- **Microsoft Antimalware** extension: Free basic antivirus for Azure VMs (Windows).
- **Microsoft Defender for Endpoint (MDE)**: Full EDR solution; integrated with Defender for Servers.
- **Vulnerability Assessment**: Qualys or Microsoft Defender VA; scans for OS and app vulnerabilities.

### Just-In-Time (JIT) VM Access

Managed by **Microsoft Defender for Cloud**:
1. Management ports (RDP 3389, SSH 22, WinRM 5985/5986) are **blocked by default** via NSG.
2. User requests access (specifying source IP, duration, port).
3. Defender for Cloud **temporarily opens** the NSG rule for the requested parameters.
4. NSG rule is **automatically removed** after the requested time.

```bash
# Request JIT access via Azure CLI
az security jit-policy initiate \
  --resource-group MyRG \
  --vm-name MyVM \
  --ports '[{"number":3389,"endTimeUtc":"2025-03-25T17:00:00.000Z","allowedSourceAddressPrefix":"203.0.113.10"}]'
```

### VM Security Best Practices

- Use **Azure Bastion** instead of exposing RDP/SSH to the internet
- Enable **JIT VM access** if direct access is required
- Apply **NSGs** to restrict traffic to VMs
- Keep VMs **patched** — use Update Manager or Automation Update Management
- Use **managed disks** with encryption (SSE with CMK or ADE)
- Apply the **principle of least privilege** for VM identities
- Enable **Boot Diagnostics** and **Guest OS diagnostics**

## 4. Azure Container Security

### Azure Kubernetes Service (AKS) Security

**Authentication and Authorization:**
- **Entra ID integration**: Use Entra ID groups to control kubectl access
- **Kubernetes RBAC**: ClusterRoles and ClusterRoleBindings for fine-grained control
- **Azure RBAC for Kubernetes**: Use Azure role assignments to control cluster access

**Network security:**
- **Network Policies**: Kubernetes-native; control pod-to-pod traffic
- **Azure CNI vs Kubenet**: Azure CNI assigns real VNet IPs to pods (better Network Policy support)
- **Private cluster**: API server accessible only via private endpoint

**Image security:**
- Use **Azure Container Registry (ACR)** to store images
- Enable **Defender for Containers** vulnerability scanning
- Use **ACR Tasks** to build and scan images in pipeline

**Pod security:**
- **Azure Policy add-on for AKS**: Enforce Kubernetes pod security standards via Gatekeeper
- Run containers as **non-root** users
- Use **read-only root filesystems** where possible

### Azure Container Registry (ACR) Security

- **Authentication**: 
  - Admin account (disabled by default — keep disabled)
  - Service principals with RBAC roles
  - Managed identities (preferred for Azure services)
  - Azure CLI token (for interactive use)

- **RBAC roles**:
  - `AcrPull`: Pull images
  - `AcrPush`: Pull and push images
  - `AcrDelete`: Delete images
  - `Owner` / `Contributor`: Full management

- **Network**: Restrict ACR to private endpoint or specific IP ranges

- **Content Trust**: Digitally sign images; only signed images can be pulled (requires Premium tier)

### Defender for Containers

Provides:
- **Vulnerability assessments** for container images (in registry and running in AKS)
- **Run-time threat detection** for AKS clusters
- **Security recommendations** for cluster configuration hardening
- **Kubernetes audit log** analysis for anomalous API calls

## 5. Azure Storage Security

### Authentication Methods

| Method | Description | Best For |
|--------|-------------|---------|
| **Storage Account Key** | Full access master key (2 keys per account) | Legacy; avoid where possible |
| **Shared Access Signature (SAS)** | Token granting scoped, time-limited access | Delegating limited access to clients |
| **Azure AD / Entra ID (RBAC)** | Identity-based access using RBAC roles | Recommended for users and services |
| **Managed Identity** | Service identity; no stored credentials | Azure services accessing storage |

### SAS Types

| Type | Description |
|------|-------------|
| **Service SAS** | Access to specific service (Blob, File, Queue, Table) and specific resource |
| **Account SAS** | Access to multiple services/resource types |
| **User Delegation SAS** | Created using Entra ID credentials; more secure than key-based SAS |

> **Exam Tip:** **User Delegation SAS** is more secure than Service/Account SAS because it uses Entra ID credentials rather than storage account keys. Use it when possible.

### Network Security

- **Firewall rules**: Restrict access to specific IP ranges or VNet subnets
- **Service Endpoints**: Route storage traffic through VNet backbone
- **Private Endpoints**: Access storage via private IP in VNet
- **Disable public blob access**: Prevent anonymous read access to blobs at account level
- **Require secure transfer (HTTPS)**: Enforce TLS; deny HTTP access

```bash
# Create a storage account with security best practices
az storage account create \
  --name mystorageacct \
  --resource-group MyRG \
  --location eastus \
  --sku Standard_LRS \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false \
  --https-only true \
  --default-action Deny \
  --bypass AzureServices Logging Metrics

# Generate a User Delegation SAS
az storage blob generate-sas \
  --account-name mystorageacct \
  --container-name mycontainer \
  --name myblob.txt \
  --permissions rwd \
  --expiry 2025-04-01T00:00:00Z \
  --auth-mode login \
  --as-user
```

### Encryption

- **Encryption at rest**: All data encrypted using AES-256 automatically (Microsoft-managed keys by default).
- **Customer-Managed Keys (CMK)**: Store encryption key in Key Vault; requires purge protection on Key Vault.
- **Encryption Scopes**: Different encryption keys for different containers/blobs.
- **Infrastructure encryption**: Double encryption (platform + additional encryption layer).
- **In transit**: Enforce HTTPS (TLS 1.2+); use `--https-only` flag.

### Defender for Storage

Detects:
- Suspicious access patterns (unusual geo, anonymous access, mass downloads)
- Malware uploaded to Blob storage (hash reputation + dynamic analysis)
- Sensitive data exposure alerts
- Phishing content hosted in storage

### Immutable Blob Storage

For compliance: WORM (Write Once, Read Many) policies on containers:
- **Time-based retention policy**: Data can't be modified or deleted for a specified period
- **Legal hold**: Indefinite immutability until legal hold is removed

## 6. Azure SQL Database Security

### Authentication

| Method | Description |
|--------|-------------|
| **SQL authentication** | Username/password stored in SQL Database; legacy approach |
| **Azure AD / Entra ID authentication** | Identity-based; recommended; supports MFA and managed identities |

> **Best practice**: Set an **Entra ID admin** on SQL Server and use Entra ID authentication. Avoid SQL authentication when possible.

### Firewall Rules

Three levels:
1. **Server-level firewall rules**: Apply to all databases on the server
2. **Database-level firewall rules**: Apply to a specific database
3. **Virtual network rules**: Allow access from specific VNet subnets (via service endpoints)
4. **Private Endpoint**: Fully private access (no public access needed)

```sql
-- Create a server-level firewall rule (in Azure portal or via CLI)
-- CLI:
az sql server firewall-rule create \
  --resource-group MyRG \
  --server mysqlserver \
  --name OfficeIP \
  --start-ip-address 203.0.113.0 \
  --end-ip-address 203.0.113.255
```

### Transparent Data Encryption (TDE)

- Encrypts SQL Database, Managed Instance, and Synapse Analytics data files at rest
- **Enabled by default** for all new Azure SQL databases
- Encryption key options:
  - **Service-managed TDE protector**: Microsoft manages the key (default)
  - **Customer-managed TDE protector (BYOK)**: Your key in Key Vault

### Always Encrypted

Encrypts sensitive column data at the **client side** — the database server never sees plaintext values for encrypted columns.

- **Column Master Key (CMK)**: Stored in a key store (Azure Key Vault or Windows Certificate Store)
- **Column Encryption Key (CEK)**: Encrypted by CMK; stored in the database
- Use cases: Credit card numbers, SSNs, medical data

```sql
-- Create encrypted column (example DDL)
CREATE TABLE Patients (
    PatientID INT PRIMARY KEY,
    SSN NCHAR(11) ENCRYPTED WITH (
        COLUMN_ENCRYPTION_KEY = CEK1,
        ENCRYPTION_TYPE = Deterministic,
        ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256'
    ) NOT NULL
);
```

**Encryption types:**
- **Deterministic**: Same value always produces same ciphertext; allows equality comparisons and joins
- **Randomized**: Same value produces different ciphertext each time; more secure but no searches on column

### Dynamic Data Masking (DDM)

Limits sensitive data exposure by masking it for non-privileged users in query results. The actual data in the database is NOT changed.

Mask types:
- **Default**: Full masking (e.g., `XXXXXXXX`)
- **Email**: Shows first character + `XXX@XXXX.com`
- **Random**: Random number in a range (for numeric columns)
- **Custom text**: Define prefix, padding, suffix

> **Exam Tip:** DDM protects against casual exposure by **masking query results** — it does NOT encrypt data and cannot prevent privileged users from seeing actual data.

### Advanced Threat Protection (Defender for SQL)

Detects anomalous activities:
- SQL injection attacks
- Anomalous access (unusual location, client, user)
- Brute-force attacks
- Suspicious database exploration

### Auditing

- Tracks database events and writes to:
  - **Azure Storage**
  - **Log Analytics workspace**
  - **Event Hub**
- **Server-level auditing** applies to all databases; database-level auditing is per-database
- Retention: Set in storage account lifecycle policy or Log Analytics retention

### Row-Level Security (RLS)

Restricts query results based on the characteristics of the user executing the query.

```sql
-- Example: Users can only see their own rows
CREATE SECURITY POLICY SalesFilter
ADD FILTER PREDICATE dbo.fn_securitypredicate(SalesRep)
ON dbo.Sales
WITH (STATE = ON);
```

## 7. Azure Cosmos DB Security

### Authentication

- **Primary/secondary keys**: Master keys for full access; **rotate regularly**
- **Resource tokens**: Scoped tokens for specific users/permissions; preferred for client apps
- **Entra ID RBAC** (Data Plane): Assign built-in Cosmos DB data plane roles
  - `Cosmos DB Built-in Data Reader`
  - `Cosmos DB Built-in Data Contributor`

### Network Security

- **IP firewall**: Restrict access to specific IP ranges
- **Service endpoints**: Route traffic via VNet
- **Private endpoints**: Fully private connectivity (recommended)
- **Disable public network access**: Force all connectivity through private endpoints

### Encryption

- All data encrypted at rest using Microsoft-managed keys (default)
- **CMK (Customer-Managed Keys)**: Configure at account creation; key stored in Key Vault
- All data encrypted in transit via TLS

### Defender for Cosmos DB

Detects:
- SQL injection attempts on Cosmos DB queries
- Unusual access patterns (anomalous data extraction, suspicious IPs)
- Data exfiltration alerts

## 8. Key Exam Facts & Practice Questions

### Must-Know Facts

1. **Azure Disk Encryption (ADE)**: Uses BitLocker (Windows) / DM-Crypt (Linux); keys in Key Vault
2. **SSE**: Platform-level encryption of managed disks; always enabled; can use CMK
3. **Key Vault Soft Delete** is now always enabled by default; **Purge Protection** must be enabled manually
4. **RBAC is the recommended** permission model for Key Vault (over access policies)
5. **User Delegation SAS** is more secure than Service/Account SAS (uses Entra ID, not storage key)
6. **TDE** is **enabled by default** on all new Azure SQL databases
7. **Always Encrypted** encrypts at client; server never sees plaintext — protects against insider threats and cloud admin attacks
8. **Dynamic Data Masking** does NOT encrypt — only masks query results; actual data unchanged
9. **JIT VM Access** managed by Defender for Cloud; temporarily opens NSG rules on-demand
10. **Defender for Cloud Secure Score**: Completing all recommendations in a security control earns its full score

### Practice Questions

**Q1.** A developer accidentally deleted an Azure Key Vault secret that is used by multiple production applications. The Key Vault has soft delete enabled with a 90-day retention period, but purge protection is NOT enabled. The developer then ran a purge operation. Can the secret be recovered?

- A) Yes, soft delete retains it for 90 days regardless
- B) No, the purge operation permanently deleted the secret
- C) Yes, Azure automatically backs up all Key Vault objects
- D) No, but the Key Vault can restore from its last backup

<details><summary>Answer</summary>
**B** — Without purge protection, a manual purge operation immediately and permanently deletes the soft-deleted secret. Purge protection prevents this by requiring the full retention period to expire. This is why purge protection should be enabled for production vaults.
</details>

---

**Q2.** An Azure SQL Database stores patient medical records including Social Security Numbers (SSNs). Compliance requires that DBAs cannot read SSN values in plaintext, but applications can query and compare SSNs. Which SQL security feature should you implement?

- A) Transparent Data Encryption (TDE)
- B) Dynamic Data Masking
- C) Always Encrypted with Deterministic encryption
- D) Row-Level Security (RLS)

<details><summary>Answer</summary>
**C** — Always Encrypted with Deterministic encryption encrypts SSNs at the client side; the database never sees plaintext, protecting against even DBAs and cloud admins. Deterministic encryption allows equality queries (needed for comparisons). TDE (A) protects files at rest but DBAs can still read data in queries. DDM (B) only masks results — DBAs can bypass masking. RLS (D) controls row access, not column encryption.
</details>

**Q3.** You need to configure an Azure VM to access Azure Key Vault without storing any credentials in the VM's code or configuration. The VM should only be able to access one specific Key Vault. What should you do? (Choose two)

- A) Enable a system-assigned managed identity on the VM
- B) Create a service principal and store the client secret in the VM's app settings
- C) Assign the Key Vault Secrets User role to the VM's managed identity on the specific Key Vault
- D) Add the VM's IP address to the Key Vault firewall
- E) Create a storage account to hold the Key Vault access key

<details><summary>Answer</summary>
**A and C** — Enable system-assigned managed identity on the VM (A), then grant that identity the Key Vault Secrets User role scoped to the specific Key Vault (C). This allows the VM to read secrets without any stored credentials.
</details>

---

**Q4.** An Azure Storage account contains sensitive financial documents. You need to ensure that a third-party vendor can upload files to a specific container for exactly 24 hours, with no ability to delete or list files. What should you provide?

- A) Storage account key 1
- B) An Account SAS with the write permission, expiring in 24 hours
- C) A Service SAS with write-only permission scoped to the specific container, expiring in 24 hours
- D) An Entra ID guest account with Storage Blob Data Contributor role

<details><summary>Answer</summary>
**C** — A Service SAS scoped to the specific container with only write permission, set to expire in 24 hours, grants exactly the required access without exposing the full account key or allowing delete/list operations.
</details>

**Q5.** Defender for Cloud shows a recommendation: "MFA should be enabled on accounts with owner permissions on your subscription." You have enabled MFA for all users via a Conditional Access policy. The recommendation still shows as unhealthy. What is the most likely reason?

- A) Conditional Access MFA is not recognized by Defender for Cloud
- B) The Conditional Access policy excludes emergency access accounts that have Owner permissions
- C) Defender for Cloud only recognizes per-user MFA settings, not Conditional Access
- D) The recommendation requires PIM to be configured, not just MFA

<details><summary>Answer</summary>
**B** — Defender for Cloud detects Owner-level accounts without MFA requirement. If emergency access ("break glass") accounts are excluded from the CA policy (a common practice) and they have Owner permissions, the recommendation will remain unhealthy.
</details>
