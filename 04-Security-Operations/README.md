## 1. Microsoft Defender for Cloud

### What It Is
Microsoft Defender for Cloud (formerly Azure Security Center + Azure Defender) is a Cloud Security Posture Management (CSPM) and Cloud Workload Protection Platform (CWPP) tool.

### Two Main Functions
| Function | Description |
|---|---|
| **CSPM (Posture Management)** | Assess and improve security posture; security recommendations; Secure Score |
| **CWPP (Workload Protection)** | Threat detection and advanced defense for specific workloads |

### Defender Plans (Workload Protection)
| Plan | Protects |
|---|---|
| **Defender for Servers** | VMs, Arc-enabled servers |
| **Defender for App Service** | App Service plans |
| **Defender for Storage** | Storage accounts |
| **Defender for SQL** | Azure SQL, SQL on VMs, PostgreSQL, MySQL, MariaDB |
| **Defender for Containers** | AKS, ACR, containerized workloads |
| **Defender for Key Vault** | Key Vault access and operations |
| **Defender for DNS** | DNS-layer threat detection |
| **Defender for Resource Manager** | ARM layer attacks |
| **Defender CSPM (enhanced)** | Advanced posture management, attack path analysis, governance |

### Free Tier (Foundational CSPM)
- Security recommendations
- Secure Score
- Compliance reports (Azure Security Benchmark only)
- Continuous assessment

### Security Alerts
- Alerts represent detected threats
- Severity: High / Medium / Low / Informational
- Each alert maps to a **MITRE ATT&CK** tactic/technique
- Alerts can be promoted to **Incidents** (correlated alerts)

### Alert Types Examples
| Alert | Description |
|---|---|
| Brute force attack on Windows | Multiple failed RDP logins |
| Suspicious process execution | Unusual parent-child process relationship |
| SQL injection attempt | Pattern matching in SQL queries |
| Anomalous access to Key Vault | Unusual access from new IP/user |
| Cryptocurrency mining | CPU anomaly suggesting mining malware |

### Exam Tips
- **Defender for Cloud** is multi-cloud — supports Azure, AWS, and GCP via Arc
- **Foundational CSPM** is free; **Defender CSPM** plan adds attack path analysis, cloud security explorer
- Alerts are generated by Defender **plans** — enable the relevant plan for protection
- **JIT VM Access** is configured through Defender for Cloud → Defender for Servers
- Know the difference between **recommendations** (posture) and **alerts** (threats)

## 2. Microsoft Sentinel

### What It Is
Microsoft Sentinel is a cloud-native SIEM (Security Information and Event Management) and SOAR (Security Orchestration, Automation, and Response) solution.

### Key Components
| Component | Description |
|---|---|
| **Data connectors** | Ingest data from various sources (Azure services, third-party, on-premises) |
| **Workspace** | Log Analytics workspace where all data is stored |
| **Analytics rules** | Detect threats using scheduled queries, ML, Microsoft security alerts |
| **Incidents** | Correlated group of related alerts |
| **Hunting** | Proactive threat hunting using KQL queries |
| **Notebooks** | Jupyter notebooks for advanced investigation |
| **Watchlists** | Reference data (e.g., list of VIPs, vulnerable assets) |
| **Threat intelligence** | Import and manage threat indicators (IOCs) |
| **Playbooks** | Automated response using Logic Apps |
| **Workbooks** | Visualization dashboards |
| **UEBA** | User and Entity Behavior Analytics — detect anomalies |

### Architecture
```
Data Sources
  → Data Connectors
    → Log Analytics Workspace (all data stored as tables)
      → Analytics Rules (KQL queries for detection)
        → Alerts → Incidents
          → Playbooks (automated response)
          → Investigation (manual)
```

### Data Connector Categories
- **Azure native**: Defender for Cloud, Azure AD, Azure Activity, Azure Firewall, NSG Flow Logs
- **Microsoft 365**: Office 365, Defender XDR, Defender for Endpoint
- **Third-party**: AWS CloudTrail, Palo Alto, Cisco, etc.
- **Syslog / CEF**: Any device that sends syslog or Common Event Format
- **REST API / Logic Apps**: Custom data ingestion

### Analytics Rule Types
| Type | Description |
|---|---|
| **Scheduled** | KQL query runs on schedule; returns results as alerts |
| **Microsoft Security** | Auto-create incidents from alerts in Defender products |
| **Fusion** | ML-based multi-stage attack detection |
| **Anomaly (UEBA)** | ML-based behavioral anomaly detection |
| **Threat Intelligence** | Match IOCs against logs |
| **NRT (Near-Real-Time)** | Scheduled rule running every minute |

### Analytics Rule Configuration
Key fields when creating a scheduled rule:
- **KQL query** — detection logic
- **Entity mapping** — map query results to Sentinel entities (account, host, IP, etc.)
- **Alert threshold** — minimum results to generate an alert
- **Event grouping** — one alert per row vs. one alert per query run
- **Query scheduling** — how often to run (default: every 5 minutes)
- **Lookback period** — how far back the query looks (default: 5 hours)
- **Suppression** — suppress duplicate alerts for a period

### KQL Essentials for Sentinel

```kql
// Find sign-in failures
SigninLogs
| where ResultType != 0
| summarize FailureCount = count() by UserPrincipalName, IPAddress
| where FailureCount > 10
| order by FailureCount desc

// Detect new service principals
AuditLogs
| where OperationName == "Add service principal"
| project TimeGenerated, InitiatedBy, TargetResources

// Alert on admin role assignments
AuditLogs
| where OperationName == "Add member to role"
| where TargetResources[0].modifiedProperties[0].newValue contains "Global Administrator"
```

### Incident Management
- Incidents are created from correlated alerts
- Properties: Severity, Status (New/Active/Closed), Owner, Evidence, Timeline, Entities
- **Investigation graph**: Visualize relationships between entities in an incident
- Closing reasons: True Positive, Benign Positive, False Positive, Undetermined

### UEBA (User and Entity Behavior Analytics)
- Builds behavioral baselines for users and entities (hosts)
- Anomaly scoring: peer comparison, temporal anomalies
- Timeline view for entity investigation
- Enriches incidents with user/entity context

### Playbooks (SOAR)
- Built on **Azure Logic Apps**
- Triggered by: alert creation, incident creation, entity trigger
- Example automations:
  - Block a user when an alert fires
  - Notify SOC team via Teams/email
  - Create a ticket in ServiceNow/Jira
  - Enrich an alert with threat intelligence

### Exam Tips
- **Sentinel = SIEM + SOAR** — know both detection (analytics rules) and response (playbooks)
- **Playbooks** use Logic Apps; **Notebooks** use Jupyter for advanced investigation
- **UEBA** requires enabling and waiting for a learning period (7 days minimum)
- **Entity pages** provide a 360-degree view of a user or host in Sentinel
- Know the difference between **Sentinel incidents** (Sentinel product) and **Defender for Cloud incidents** (Defender product) — both can feed into Sentinel
- **Content Hub** in Sentinel provides packaged solutions (data connectors + analytics rules + workbooks) for specific use cases

## 3. Log Analytics and Azure Monitor

### Log Analytics Workspace
- **Foundation** for Sentinel, Defender for Cloud, and Azure Monitor
- All log data stored as tables in the workspace
- KQL used for querying

### Key Tables for Security
| Table | Data Source |
|---|---|
| `SigninLogs` | Entra ID sign-in logs |
| `AuditLogs` | Entra ID audit events |
| `SecurityAlert` | Alerts from Defender products |
| `SecurityEvent` | Windows security events |
| `Syslog` | Linux syslog events |
| `AzureActivity` | Azure control plane operations |
| `AzureFirewallApplicationRule` | Azure Firewall logs |
| `AzureFirewallNetworkRule` | Azure Firewall network rule logs |
| `StorageBlobLogs` | Azure Storage blob operations |
| `SqlSecurityAuditEvents` | Azure SQL audit events |
| `CommonSecurityLog` | CEF format events (third-party) |
| `ThreatIntelligenceIndicator` | Threat intelligence IOCs |
| `IdentityInfo` | UEBA enriched user data |
| `BehaviorAnalytics` | UEBA behavioral data |

### Azure Monitor Alerts
- **Metric alerts**: Based on resource metrics (CPU %, inbound bytes)
- **Log alerts**: KQL queries on Log Analytics; trigger when threshold crossed
- **Activity log alerts**: Azure control plane events (e.g., delete resource group, change policy)

### Diagnostic Settings
- Configure **where** Azure resource logs and metrics are sent:
  - Log Analytics Workspace (for querying/alerting)
  - Storage Account (for archival)
  - Event Hub (for streaming to third-party SIEM)
- Must be enabled per resource — not enabled by default

### Azure Activity Log
- Records all **management plane** operations in your subscription
- Who did what, to which resource, at what time
- Retained for **90 days** by default; send to Log Analytics for longer retention

### Exam Tips
- **Diagnostic settings** must be explicitly configured per resource
- **Activity Log** covers control plane (ARM); **Resource logs** cover data plane operations
- Know the key log tables for security investigation
- Data retention in Log Analytics: configurable (30 days free, up to 730 days paid; archive up to 12 years)

## 4. Security Policies and Compliance

### Azure Policy
- Define **rules** (policy definitions) that Azure resources must comply with
- Effects:
  | Effect | Behavior |
  |---|---|
  | **Audit** | Log non-compliant resources; don't block |
  | **Deny** | Block creation/modification of non-compliant resources |
  | **DeployIfNotExists** | Auto-deploy a resource if not present |
  | **AuditIfNotExists** | Audit if a related resource doesn't exist |
  | **Modify** | Add/update/remove resource tags or properties |
  | **Append** | Add fields to a resource during create/update |
  | **Disabled** | Turn off policy evaluation |

### Policy Initiative
- Group of multiple policy definitions
- Azure Security Benchmark is an initiative (collection of policies)
- Custom initiatives can be created

### Regulatory Compliance
- Defender for Cloud maps controls to regulatory standards:
  - Azure Security Benchmark (default)
  - CIS, NIST 800-53, PCI DSS, HIPAA, ISO 27001, SOC 2, and more
- Compliance dashboard shows overall compliance score per standard

### Exam Tips
- **Deny** effect blocks non-compliant resources **at creation**; **Audit** only logs
- **DeployIfNotExists** automates remediation (e.g., auto-deploy Log Analytics agent)
- Security Center/Defender for Cloud uses **initiatives** — a set of policies applied to subscriptions

## 5. Azure Security Benchmark and Secure Score

### Azure Security Benchmark (ASB)
- Microsoft's security best practice recommendations for Azure
- Aligned with CIS Controls and NIST SP 800-53
- Organized by control domains: Identity, Network, Data, Application, Endpoint, etc.
- Drives the default recommendations in Defender for Cloud

### Secure Score
- A percentage reflecting how well you've implemented security recommendations
- Each recommendation has a **point value**
- Completing a recommendation increases your score
- Score = (Points achieved / Total possible points) × 100%

### How to Use Secure Score
1. Review recommendations sorted by impact
2. Prioritize **Quick fix** recommendations
3. Assign recommendations to owners with due dates (**governance rules** in enhanced CSPM)
4. Track progress over time

### Exam Tips
- Secure Score **only** reflects security controls, not threats/incidents
- A score increase = security posture improvement
- Not all recommendations have equal weight — focus on highest impact first

## 6. Threat Intelligence

### Microsoft Defender Threat Intelligence (MDTI)
- Repository of threat intelligence data: IOCs, threat actor profiles, vulnerability data
- Integrated into Sentinel, Defender XDR, Defender for Cloud

### Threat Intelligence in Sentinel
| Feature | Description |
|---|---|
| **TAXII feed ingestion** | Import IOCs from STIX/TAXII threat intel feeds |
| **Microsoft TI connector** | Import Microsoft's own threat intelligence |
| **ThreatIntelligenceIndicator table** | All IOCs stored here in Sentinel workspace |
| **Threat Intelligence blade** | Manage indicators, view relationships |
| **Analytics rules** | Match ingested IOCs against logs (TI Map rules) |

### IOC Types
- **IP addresses**: Malicious source/destination IPs
- **Domain names**: Malicious FQDNs
- **URLs**: Malicious URLs
- **File hashes**: Malicious file signatures (MD5, SHA1, SHA256)
- **Email addresses**: Malicious senders

### Exam Tips
- TI data in Sentinel is stored in the `ThreatIntelligenceIndicator` table
- **STIX/TAXII** is the standard format/protocol for sharing threat intelligence
- Enable the **Microsoft Threat Intelligence** data connector for automatic IOC ingestion
- **Threat Intelligence Matching Analytics** rules automatically correlate logs with IOCs

## 7. Incident Response and Automation

### Incident Response Lifecycle (NIST)
1. **Preparation** — Policies, playbooks, training
2. **Detection & Analysis** — SIEM alerts, investigation
3. **Containment** — Isolate affected systems
4. **Eradication** — Remove threat
5. **Recovery** — Restore normal operations
6. **Post-Incident Activity** — Lessons learned, reporting

### Azure Tools for Each Phase
| Phase | Azure Tools |
|---|---|
| Preparation | Sentinel playbooks, Logic Apps, Runbooks |
| Detection | Sentinel analytics rules, Defender for Cloud alerts |
| Analysis | Sentinel investigation graph, entity pages, KQL hunting |
| Containment | Playbook: block user/IP, isolate VM (MDE) |
| Eradication | MDE: remove malware, remediate vulnerabilities |
| Recovery | Backup/Restore, Azure Site Recovery |
| Post-Incident | Sentinel incident timeline, audit logs |

### Automation in Sentinel

#### Automation Rules
- Lightweight, fast response to incidents and alerts
- Can: assign owner, change severity/status, add tags, run playbook
- Applied based on conditions (incident title contains X, alert provider is Y)
- Run **before** playbooks — useful for triage

#### Playbooks (Logic Apps)
- Full workflow automation
- Triggered by: alerts, incidents, or entity actions
- Common patterns:
  ```
  Incident created → Get user entity → Check Entra ID risk →
  If high risk → Block sign-in → Post to Teams → Update incident
  ```

### Azure Automation
- **Runbooks**: PowerShell/Python scripts for automation
- **Update Management**: Automate OS patching
- **DSC (Desired State Configuration)**: Enforce VM configuration state

### Exam Tips
- **Automation rules** run first (fast, low-code), then **playbooks** (complex, full Logic App)
- Know the difference between Sentinel **playbooks** (Logic Apps) and Azure Automation **runbooks**
- Incident lifecycle: New → Active → Closed (True Positive / False Positive / Benign Positive)
- **MDE Live Response** allows real-time investigation and containment from Defender portal

## Key Exam Tips

1. **Defender for Cloud = CSPM + CWPP** — free tier for posture; paid plans for workload protection
2. **Sentinel = SIEM + SOAR** — analytics rules detect, playbooks respond
3. Know the **KQL basics** — you'll likely face log query scenario questions
4. **Playbooks** = Logic Apps; **Notebooks** = Jupyter; **Runbooks** = Azure Automation
5. **Automation rules** are fast/lightweight; **playbooks** are for complex workflows
6. Diagnostic settings must be **explicitly configured** per resource to ingest logs
7. **UEBA** builds baselines over time; don't expect instant results
8. **Secure Score** measures posture; it does NOT reflect active incidents
9. **Activity Log** = control plane (90 days default); configure to Log Analytics for longer retention
10. Policy **Deny** effect blocks at creation; **Audit** only logs; **DeployIfNotExists** auto-remediates
